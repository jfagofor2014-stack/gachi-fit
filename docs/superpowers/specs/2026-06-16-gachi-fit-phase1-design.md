# GACHI-FIT Phase1 設計書

## コンセプト
中・上級トレーニー向けの「感覚同期型」トレーニング最適化アプリ。既存アプリが「量」を記録するのに対し、GACHI-FIT は「質（感覚）」をデータ化する。ジムの現場で片手・1-Tap で爆速操作できる UX を最優先する。

## ターゲット
- 中級者：フォーム習得後、伸び悩みを打破したい層
- 上級者：大会出場を目指し、ミリ単位のフォーム変化を記録したい層

## スコープ（Phase1）
コア記録機能の構築。AI インサイト（Phase3）・体形比較（Phase3）は対象外だが、Sensory データ構造は将来 Gemini 連携できるよう整える。

## 技術スタック
- Vanilla JS（ESモジュール分割）＋ HTML / CSS
- PWA（manifest.json + Service Worker でオフライン動作・ホーム画面追加）
- IndexedDB によるローカル永続化（サーバ不要）

## 画面構成（SPA・下部タブ切替）
1. **ホーム** — 今日のメニュー候補、直近PR、進行中インターバルタイマー
2. **記録（Workout）** — メニュー選択 → セットごとに 1-Tap 記録
3. **メニュー管理** — 種目・部位細分化、セットパターン保存、意識ポイントプリセット
4. **履歴／PR** — 種目別の最高記録と推移

## コア機能

### ハイパーカスタムメニュー
- 部位を細分化して種目登録（例：背中上部 / 広がり / 厚み）
- 種目ごとに「意識ポイント」プリセットを登録
- セットパターン（ピラミッド / ドロップ / レストポーズ等）を保存・再利用

### セット記録 + RM自動計算
- 重量 × 回数を入力
- 推定1RM = `Weight × (1 + reps / 30)`（Epley式）を即時表示
- 種目選択時に過去PRと「今回目標」を常時固定表示

### Sensory Log（定型化メモ）
各セットにワンタップで以下を記録：
- 腹圧保持：1–5段階
- 対象筋への負荷：1–5段階
- 可動域 ROM：フル / 部分 / チーティング

セット品質スコア：`Score = Σ ( V_volume × W_form + I_intensity × W_core )`
（V=対象筋負荷, I=ROM係数, W_form/W_core=重み付け。Phase1 では固定重みで算出し、スコアを履歴に保存）

### インターバルタイマー
- セット完了で自動スタートする大型カウントダウン
- 手が震えていても押せる大型UIコンポーネント

## データモデル（IndexedDB）

- **exercises**: `{ id, name, bodyPart, cuePresets: string[], setPattern }`
- **workouts**: `{ id, date, note }`
- **sets**: `{ id, workoutId, exerciseId, weight, reps, estimated1RM, targetWeight, createdAt }`
- **sensoryLogs**: `{ id, setId, core(1-5), muscleLoad(1-5), rom('full'|'partial'|'cheating'), score }`

PR は sets から種目ごとの最大 estimated1RM を算出（派生データ、保存しない）。

## デザイン方針
- 黒背景 × ライムイエロー（#CCFF00系）アクセント、要件定義書のトーン踏襲
- 大型タップターゲット、片手操作前提、現場でも視認できる高コントラスト

## エラーハンドリング
- 数値未入力／不正値はセット保存をブロックしインライン警告
- IndexedDB 失敗時はトースト通知し操作を継続不能にしない

## テスト
- スコア算出・1RM計算など純粋ロジックは関数を分離しユニットテスト可能にする
- 主要フロー（メニュー作成→記録→PR反映）を手動で動作確認
